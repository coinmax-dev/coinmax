# CoinMax V4 系统链路文档

## 一、总览

```
用户 USDT (BSC)
  → PancakeSwap V3 Swap → USDC
  → Engine V2 接收
  → 铸造 cUSD (链上记账)
  → 存入 ERC4626 Vault (cUSD 质押)
  → 铸造 MA 锁仓
  → 每日产出 cUSD 利息 (链上对账)
  → 数据库记录收益分配
  → 用户提现 → 铸造 MA → 销毁比例 → 线性释放
```

---

## 二、入金链路

### 2.1 用户存入金库

```
用户钱包 (BSC)
  │ USDT
  ▼
PancakeSwap V3 (0x92b7 池子)
  │ Swap USDT → USDC
  ▼
Engine V2 钱包 (0xDd66)
  │ 接收 USDC
  │
  ├─→ [链上] 铸造 cUSD (1:1 USDC 记账)
  │     ▼
  │   ERC4626 Vault 合约 (0x08a2)
  │     │ 存入 cUSD → 用户获得 vault shares
  │     │ 铸造 MA 代币 → 锁仓 (Release 合约)
  │     ▼
  │   锁仓 MA (按计划周期)
  │
  └─→ [数据库] 记录用户存入金库
        │ 用户ID、金额、计划类型、入金时间
        ▼
      触发后续收益计算
```

## 三、每日收益结算 (Cron 12:00 SGT)

### 3.1 金库收益

```
每日结算触发 (12:00 SGT)
  │
  ▼
[链上] ERC4626 根据质押 cUSD 周期产出每日利息 cUSD
  │     (仅作为链上对账记录，不直接发放)
  │
  ▼
[数据库] 记录每日收益
  │
  ├─→ 待释放余额 → 增加「金库收益」
  │     金库收益(MA) = 本金 × 日利率 / MA价格
  │
  ├─→ 触发「直推奖励」(详见 3.2)
  │
  └─→ 触发「团队奖励」(详见 3.3)
```

### 3.2 直推奖励 (10%)

```
用户A 产生金库日收益 = X MA
  │
  ▼
检查 用户A 的直接推荐人 B
  │
  ├─ B 是否已存入 ≥ $100 金库？
  │   ├─ 否 → 跳过，不发放
  │   └─ 是 ↓
  │
  ▼
用户B 待释放余额 → 增加「经纪人奖励 - 直推收益」
  直推收益(MA) = X × 10%
```

### 3.3 团队奖励 (级差 + 同级 + 越级)

```
用户A 产生金库日收益 = X MA
  │
  ▼
沿安置推荐树 (placement tree) 向上遍历
  │
  ▼
对每个上线节点，检查:
  1. 是否已存入 ≥ $100 金库？(否则跳过，继续向上)
  2. 获取该上线的等级和级差比例

  ┌─────────────────────────────────────────────┐
  │  等级    级差比例                              │
  │  V1      5%                                   │
  │  V2      10%                                  │
  │  V3      15%                                  │
  │  V4      20%                                  │
  │  V5      25%                                  │
  │  V6      30%                                  │
  │  V7      50%                                  │
  └─────────────────────────────────────────────┘
```

#### 3.3.1 级差奖励 (无限代)

```
计算: 当前上线级差比例 - 已分配的最高比例 = 差额比例
  │
  差额 > 0 ?
  ├─ 是 → 该上线获得: X × 差额比例
  │       记录: 待释放余额 → 经纪人奖励 → 级差奖励
  │       更新已分配最高比例
  └─ 否 → 不获得级差（进入同级/越级判断）
```

**示例:**
```
用户A (V0) 产生 100 MA 日收益

  ↑ 上线C (V2, 级差10%)
    级差 = 10% - 0% = 10% → 获得 100×10% = 10 MA

  ↑ 上线D (V4, 级差20%)
    级差 = 20% - 10% = 10% → 获得 100×10% = 10 MA

  ↑ 上线E (V6, 级差30%)
    级差 = 30% - 20% = 10% → 获得 100×10% = 10 MA
```

#### 3.3.2 同级奖励 (最多2代)

```
条件: 当前上线等级 = 前一个已分配上线的等级 (同级)

计算: X × 该级别的级差比例 × 10%

  记录: 待释放余额 → 经纪人奖励 → 同级奖励
  计数: 同级已分配次数 +1
  限制: 最多2代同级
```

**示例:**
```
  ↑ 上线F (V4) ← 前一个分配者也是 V4 → 同级!
    同级奖励 = 100 × 20%(V4级差) × 10% = 2 MA
    (同级第1代)

  ↑ 上线G (V4) ← 又遇到 V4 → 同级!
    同级奖励 = 100 × 20% × 10% = 2 MA
    (同级第2代，达到上限)

  ↑ 上线H (V4) ← 还是 V4
    → 已达2代上限，跳过
```

#### 3.3.3 越级奖励 (最多2代)

```
条件: 当前上线级差比例 < 已分配的最高比例 (被越级)

计算: X × 该上线的级差比例 × 5%

  记录: 待释放余额 → 经纪人奖励 → 越级奖励
  计数: 越级已分配次数 +1
  限制: 最多2代越级
```

**示例:**
```
  已分配最高: V6 (30%)

  ↑ 上线I (V3, 级差15%) ← 15% < 30% → 被越级!
    越级奖励 = 100 × 15%(V3级差) × 5% = 0.75 MA
    (越级第1代)
```

### 3.4 收益分类汇总

```
待释放余额
  │
  ├─ 金库收益          ← 每日金库利息产出的 MA
  │
  ├─ 经纪人奖励
  │   ├─ 直推收益      ← 直推下线日收益 × 10%
  │   ├─ 级差奖励      ← 安置树级差 (无限代)
  │   ├─ 同级奖励      ← 同等级的级差 × 10% (最多2代)
  │   └─ 越级奖励      ← 被超越等级的级差 × 5% (最多2代)
  │
  └─ 节点收益          ← 符合标准的节点激活后产生
                         只有已解锁部分计入待释放余额
                         (MINI 锁仓需达标, MAX 即时释放)
```

## 四、节点系统

### 4.1 节点购买

```
用户 USDT → Swap USDC → Engine → Vault
  │
  ├─ MAX 节点: $600 USDC → 铸造 $6,600 cUSD (11倍配资)
  │   120天周期, 0.9%日利率
  │
  └─ MINI 节点: $100 USDC → 铸造 $1,100 cUSD (11倍配资)
      90天周期, 0.9%日利率
```

### 4.2 节点收益

```
节点每日产生 cUSD 利息 (链上对账)
  │
  ├─ MAX 节点: 收益即时解锁 → 待释放余额 - 节点收益
  │
  └─ MINI 节点: 收益锁仓
      │ 需要达到 milestone 才能解锁
      │ 达标 → 解锁到待释放余额
      │ 未达标 → 继续锁仓或销毁
      ▼
    节点收益 (已解锁部分) → 待释放余额 - 节点收益
```

### 4.3 节点激活等级

| 等级 | 金库存款要求 | MINI推荐要求 | 适用节点 |
|------|------------|-------------|---------|
| V1 | $100 | 3个MINI | MAX |
| V2 | $300 | 0 | MAX/MINI |
| V3 | $500 | 0 | MAX/MINI |
| V4 | $600 | 0 | MAX/MINI |
| V5 | $800 | 0 | MAX |
| V6 | $1,000 | 0 | MAX |

## 五、用户提现 (待释放余额 → MA → 线性释放)

### 5.1 提现流程

```
用户触发提现 (从待释放余额)
  │
  ▼
数据库计算可提现 MA 总额
  │ = 金库收益 + 经纪人奖励(直推+级差+同级+越级) + 节点收益(已解锁)
  │
  ▼
用户选择「收益分成比例」
  │
  ├─ 比例 A: 销毁 0% MA → 释放周期长
  ├─ 比例 B: 销毁 5% MA → 释放周期中
  ├─ 比例 C: 销毁 10% MA → 释放周期短
  └─ 比例 D: 销毁 20% MA → 最快释放
  │
  ▼
[链上] 铸造 MA 代币 (Release 合约)
  │
  ├─ 销毁部分: 立即 burn MA → 影响 Oracle 价格 (供应减少)
  │
  └─ 剩余部分: 线性释放计划
      │ 平均分配到释放天数
      │ 每日自动释放到用户钱包
      ▼
    用户钱包收到 MA 代币
```

### 5.2 收益分成比例

| 档位 | 销毁比例 | 释放周期 | 实际获得 |
|------|---------|---------|---------|
| A | 0% | 最长 | 100% MA (慢) |
| B | 5% | 较长 | 95% MA |
| C | 10% | 较短 | 90% MA |
| D | 20% | 最短 | 80% MA (快) |

## 六、闪兑 (MA → USDT)

### 6.1 用户闪兑流程

```
用户持有 MA 代币 (BSC)
  │
  ▼
FlashSwap 合约: requestSwap(maAmount)
  │ 读取 Oracle 价格
  │ 计算 USDT 金额 = MA数量 × Oracle价格
  │ burn MA → 供应减少 → Oracle 价格调整
  │ 发出 SwapRequested 事件
  │
  ▼
Engine 监听事件 (自动处理):
  │
  │ [资金来源] 0x3869 (ARB Trading钱包)
  │   │ USDC 跨链回 BSC
  │   ▼
  │ Engine (0xDd66) 收到 USDC
  │   │
  │   ▼
  │ PancakeSwap V3 (Pool 0x92b7)
  │   │ Swap USDC → USDT
  │   ▼
  │ USDT 转给用户
  │
  ▼
FlashSwap 合约: fulfillSwap() → 标记完成
```

### 6.2 链上可见交易记录

```
TX1 (用户发起):
  MA Transfer: 用户 → 0x000...000 (销毁)

TX2 (Engine 执行):
  USDC Transfer: Engine(0xDd66) → PancakeSwap Pool(0x92b7)
  USDT Transfer: Pool(0x92b7) → Engine(0xDd66)
  USDT Transfer: Engine(0xDd66) → 用户
```

### 6.3 Oracle 价格机制

```
价格 = max(地板价, max(背书价, 增值价))

背书价 = 全部 cUSD 总量 / MA 流通量
增值价 = 基础价 × (1 + 0.1%)^天数
地板价 = $0.90 (绝对最低)

MA 铸造 → 流通量增加 → 背书价下降 (但 cUSD 同步增加，比例稳定)
MA 销毁 → 流通量减少 → 背书价上升 → 价格支撑
```

## 七、资金跨链分配

### 7.1 USDC 跨链 (BSC → ARB)

```
USDC Receiver (0xe193, BSC)
  │ 累积用户入金的 USDC
  ▼
thirdweb Bridge → ARB
  │
  ▼
ARB 分配:
  ├─ 60% → 0x3869 (Trading + 闪兑储备)
  │         │ 部分 → HyperLiquid (0xd6e5) AI交易
  │         └ 部分 → 留存做闪兑回流
  │
  ├─ 20% → 0x85c3 (Investor)
  ├─ 12% → 0x1C4D (Marketing)
  └─  8% → 0xDf90 (Operations)
```

### 7.2 闪兑回流 (ARB → BSC)

```
0x3869 (ARB)
  │ USDC
  ▼
thirdweb Bridge → BSC
  │
  ▼
Engine (0xDd66, BSC)
  │ USDC → PancakeSwap → USDT
  ▼
FlashSwap 合约 / 直接转用户
```
